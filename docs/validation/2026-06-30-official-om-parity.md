`1000,975,950,925,900,850,800,750,700,650,600,550,500,400,300,200`

GFS run returned non-null values for this level set instead:

`pressureLevelsToKeep` is:

local deployment generated custom intermediate pressure levels that the public

candidate wrapper cleanup.

Representative raw-variable mismatch at `lat=4.06, lon=74.9`, GFS run
`2026-06-29T06:00`, forecast hour `2`:

- local `.om` API `wind_v_component_10m`: `-2.7`
- public Open-Meteo single-run API `wind_v_component_10m`: `-2.8`

The source GRIB messages were then compared directly:

- original NOAA/AWS full GRIB message:
  `gfs.t06z.sfluxgrbf002.grib2`, `VGRD:10 m above ground:2 hour fcst`
- original packing: `grid_complex_spatial_differencing`
- original nearest decoded value: `-2.7500000000`
- NOAA `filter_gfs_sflux.pl` regional response packing: `grid_simple`
- NOAA filtered regional decoded value at the same point: `-2.7499987793`

This crosses the one-decimal rounding boundary. Open-Meteo generated from the
original full GRIB rounds to `-2.8`; the NOAA filter regional response rounds to
`-2.7`.

Root cause:

- NOAA filter subregion transport repacks GRIB messages and can change exact
  source floats.
- Those source-float differences propagate into raw wind components and then
  into derived wind speed, wind direction, apparent temperature, and any other
  derived variables.
- This is outside weather-code, interpolation, UI mapping, and Open-Meteo API
  reader logic.

Required correction:

- do not patch rounding, interpolation, weather-code, or derived-variable logic
- do not download official processed `.om`
- do not use NOAA filter endpoints for GFS parity
- download the selected original GRIB messages through the upstream indexed
  GRIB byte-range downloader
- decode the original full-grid message through Open-Meteo/eccodes
- crop the decoded array to the configured China-region grid before writing the
  local `.om`

This changes only the source-download/region boundary needed for China-region
production. The Open-Meteo weather readers, interpolation, weather-code mapping,
derived-variable formulas, API controller behavior, and `.om` writers remain on
the upstream path.

## Fifth Finding: Cropped Grid Coordinates Must Use Full-Grid Arithmetic

After switching away from NOAA filter endpoints, the first GFS013 50-point gate
mostly matched official Open-Meteo, but `direct_normal_irradiance` still had
small `0.1` differences on a few points.

Validated reports:

- `/tmp/wfs-validation-gfs013-50x50-20260629T221252Z/batch-01-gfs.json`
- `/tmp/wfs-validation-gfs013-floatgrid-50x50-20260629T225710Z/batch-01-gfs.json`

Root cause:

- the decoded values matched official values
- the remaining differences came from tiny latitude differences produced by
  computing cropped-grid coordinates as `regionalLatMin + localY * dy`
- official Open-Meteo computes coordinates from the full global grid as
  `globalLatMin + globalY * dy`
- floating-point addition is not associative, and solar-derived variables are
  sensitive enough for that difference to cross a one-decimal rounding boundary

Required correction:

- keep a `RegionalRegularGrid` wrapper for GFS cropped data
- use Open-Meteo's original `RegularGrid` for `findPoint`,
  `findPointInterpolated`, and `getCoordinates`
- map only the resulting full-grid x/y index back into the cropped local index

This is a regional-crop coordinate adaptation only. It does not change
interpolation, solar, weather-code, or derived-variable formulas.

Verification after this correction:

  coordinates and values

## Sixth Finding: GFS Crops Need A One-Cell Computational Halo

The next GFS013 500-point gate used point offset `150` so it did not reuse the
previous 50+100 points.

Report:

- `/tmp/wfs-validation-gfs013-regionalgrid-500x50-offset150-20260630T002638Z`
- batches 1-9 passed
- batch 10 failed at one point near the configured northeast crop boundary:
  `lat=57.955385, lon=139.946154`
- total checked before failure: `1375000`
- failure groups: temperature-like variables and derived variables depending on
  temperature/elevation correction

The same request with `elevation=nan` showed raw weather data matched official
Open-Meteo. The mismatch came from the static model elevation file `HSURF.om`:

- local cropped-grid model elevation at the boundary grid point: `790m`
- official full-grid model elevation at the same grid point: `763m`
- NOAA original `HGT:surface` GRIB value at that grid point:
  `763.0971484375`

Temporary validation with a manually expanded crop
(`WEATHER_REGION_RIGHT_LON=140.2`, `WEATHER_REGION_TOP_LAT=58.2`) generated a
0-4h smoke dataset and matched official values for the failing boundary point:

- default elevation output matched official
- `elevation=nan` model elevation matched official (`763m`)
- temperature, surface temperature, soil temperature, humidity, and wind matched
  official

Root cause:

- Open-Meteo point selection and elevation correction operate on grid
  neighborhoods and static model elevation.
- If the production region is cropped exactly to the user-facing boundary, valid
  in-region points near the boundary can become edge cells in the local `.om`
  database.
- Those edge cells do not behave like the same cells inside the official full
  grid for `HSURF`/elevation correction.

Required correction:

- include a fixed one-cell computational halo for GFS source decoding and GFS
  regional grid definitions
- do not expose a runtime switch for world-vs-China or halo-vs-no-halo behavior
- keep CAMS unchanged until a CAMS-specific boundary need is proven

This is still part of the required China-region source-data boundary. It does
not change Open-Meteo weather formulas, interpolation formulas, weather-code
mapping, or API formatting.
